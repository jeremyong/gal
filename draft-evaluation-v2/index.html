Expression Tree Evaluation

This section documents the general algorithm used to take a set of inputs and evaluate them to produce a set of results given a user-defined set of operations. To describe the method more compactly, we introduce the following notation:

An entity for which its parameters are known as a member of the set of all runtime inputs.
The corresponding entity expressed in indeterminate form1.
An intermediate result of the fuller computation as a function of some subset of and .
The result of a requested evaluation as a function of a subset of the input and intermediate expressions.
A user defined function which specifies how to map from a set of indeterminate inputs to a set of indeterminate results.
The computation's execution strategy as a map from the runtime-valued inputs to the runtime-valued outputs via a set of intermediate results. The map is induced by the user-defined map which is specified literally as an anonymous lambda function (hence the notation). In words, a lambda defines what the user wishes to compute, and big-lambda defines how we intend on satisfying that wish.

As you can see, we adopt the convention that a bar overhead a symbol denotes a compile-time indeterminate expression, whereas the absense of the bar denotes a runtime-valued quantity. The goal is to find in a manner that provides strict upper bounds on memory usage and compiler runtime (within a factor of the complexity of the worst-case member of ). The challenges involved in identifying an optimal are numerous:

  • Reification of the final results from and to must happen in the final pass jointly or the compiler will fail to inline the code.
  • Before expanding the expression tree, the complexity isn't fully known, and as such, the memory required to allocate space to accommodate is similarly unknown.
  • Processing an evaluation to produce an intermediate result generally increases as (in words, the size of a resultant indeterminate expression scales with the product of the sizes of the inputs). Mismanaging this (i.e. going straight from to ) can quickly explode and make compilation slow or impossible on finite hardware.
  • By the same token, attempting too hard to identify an optimal at compile-time is in and of itself an expensive proposition.
  • It is often the case that sub-expressions are dependencies of multiple results. In such a case, we would like to store the result as a . Even if the compiler can ultimately eliminate such an expression in a late-stage optimization pass, such elimination is not guaranteed depending on the complexity of and also places an unnecessary burden on the compiler.
  • Perhaps most-restrictive of all is that the user-defined computation supplied as an anonymous lambda function cannot be constructed or invoked as it is itself not treated by the compiler as a constant expression.2 This restriction implies that the expression trees cannot be described using flattened structs and arrays and must be encoded as types instead (i.e. the information the user can provide is based on the std::invoke_result_t of the input algorithm).

For the purposes of GAL, we relax the restriction that is necessarily optimal, and instead employ a number of heuristics to get a close enough result that is, at the very least, guaranteed to compile on most machines and produce accurate results. Expression tree construction

Input

The inputs are the set and the user-defined function .

Algorithm

  1. During expression tree construction, each operation node computes the hash of its inputs and the operation. In the case of a unary operation, the hash sum has two arguments (input hash, and operation hash), and in the case of a binary operation, the hash sum has three arguments. In this manner, any two subtree hashes can be compared for equality in constant time. For leaf nodes, the hash is fully determined by the terms present in the node's multivector, the indeterminate count, and the id of the first indeterminate.
  2. Each tree maintains at a top level, a list of additional trees representing the set of intermediate subexpressions identified.
  3. Each node maintains a hash sum of itself and all of its children, making the tree a special form of a Merkle tree.3
  4. Additionally, the expression tree maintains a both continuously updated bloom filter.4 The advantages of the bloom filter is that it occupies constant memory which is extremely desirable for compile-time computation.
  5. When a binary operation joins two nodes, it is possible that a child or the root of the LHS matches a child or a root of the RHS.
    1. Without loss of generality, swap the LHS and RHS if necessary so that the number of nodes in the LHS is less than the number of nodes in the RHS.
    2. To detect a common subexpression, traverse the LHS and perform an existence test for each node on the bloom filter of the RHS.5
    3. If a match is found, we promote the subtree to the top level , and replace both matching nodes with a placeholder node . The node is given an identifier corresponding to the index of the subtree in the top-level list.
    4. Crucially, the hash of is not changed.
    5. The top level lists of and themselves need to be merged which will require a renumbering. During the merge, the intermediates are kept sorted in hash order so that identical trees are deduplicated easily. The result is the a remapping of the top level list to .
  6. A unary operation that is marked transcendental is immediately promoted to an intermediate .
